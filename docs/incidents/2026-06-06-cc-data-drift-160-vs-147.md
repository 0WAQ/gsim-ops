# cc_all 数据 160 vs 147 指纹比对报告

**日期**: 2026-06-06
**报告人**: wbai
**范围**: 上海中信 IDC (147) vs 北京 IDC (160) 上的 `cc_all/*.npy`, T 轴切到 ≤ 20241231
**严重级别**: MEDIUM (含 1 处 schema 不一致 + 1 处 2 年数据缺口)
**当前状态**: 待决, 本地不动手, 优先级低于代码漂移 (见 `2026-06-06-gsim-code-drift-three-sites.md`)
**关联**: 跟代码漂移事件是同一审计批次, 但数据这边比代码乐观, 大部分文件等价

---

## 1. 摘要

对 cc_all 内 2106 个共同 `.npy` 数据文件 (2D float64 / int8, 切到 ≤ 20241231 的部分) 做了**聚合指纹比对**:

- **91% 文件等价** (1915 / 2106), sum + nan_count 沿 N 轴归约后跨机 `np.allclose(rtol=1e-5)` 通过
- **少量真实问题** 集中在: pwang 历史缺 2 年、Basedata/st.npy 跨机 dtype 不一致、fore_* 财务预期的 build 时点差异
- **代码漂移影响有限**: 即使刚才发现 `umgr_all.py` 等共名差异, 数据本身大部分等价, 说明算法层差异主要是性能/特性, 不是数值计算

---

## 2. 审计方法

### 2.1 设计原则

只看 `.npy` 数据本身, 不比 `.meta` / `mtime` / size。容差 `rtol=1e-5` 吸收浮点累积差异 (BLAS 多线程顺序等)。不传原始数据, 只跨机传 ~40MB 指纹。

### 2.2 指纹定义

对每个 `(T, N)` 2D `.npy`, 沿 N 轴归约得 `(T,)` 向量:

```python
sum = np.nansum(arr, axis=1)         # 当日数据量级
nan = np.isnan(arr).sum(axis=1)      # 当日 NaN 数
shape, dtype                          # 形状 + 类型 metadata
```

### 2.3 切片

T 轴只取 ≤ 20241231 段 (cutoff_idx = 3657, 从 cc_all 整长 3997 切下)。25 年的部分不比, 避开"各地 cutoff 不同步"和"yifei L2 实时增量"两类干扰。

### 2.4 跳过

| 类别 | 目录 | 原因 |
|---|---|---|
| L2 系列 | `cn_equity`, `cn_equity_feature`, `cn_equity_feature_5min`, `realtime` | 147 已知缺 25 前 L2, 不可比 |
| 3D | `Interval5m` | v1 不处理 3D `(T, K, N)` |
| 切片 | `delta` | 奇怪 shape (per-date) |
| universe / 元数据 | `__universe`, `ALL*`, `FULL`, `HS300`, `ZZ500`, `ZZ1000`, `ipo`, `TOPxxxx` | 用户要求只比数据不比 mask |

### 2.5 已知盲区 (v1 局限)

- **3D 财务一致预期** (`ashareconsensusrollingdata_*` shape `(T, K, N)` K=3/12) — 81 个 `aindexeodprices/*` 一维序列也漏 (size=31976 不整除 N\*8), v2 加
- **沿 N 归约可能掩盖逐股偏差** — 多股偏差在 sum 上抵消会漏报; v2 加抽样 + min/max/std 多指标
- **universe mask 漂移** — 用户当前不比, 但可能存在 ST/退市状态对齐差异

## 4. 发现分类 (按严重度)

### 4.1 🚨 严重: `DmgrPwang` industry 派生 feature 在 160 缺 2 年历史

**41 个文件全是同一模式**:
- 160 上 `DmgrPwang_industry_equ_fancy_factors/*.npy` 数据从 **idx=743 (20130104)** 开始
- 147 上从 **idx=256 (20110104)** 开始
- **缺口**: 20110104 ~ 20121231, **487 天连续**, 不是零散漏跑
- 方向单一: 160 缺, 147 全, 没有反向

| 样例 | 160 有效天 | 147 有效天 | 160 缺的范围 |
|---|---|---|---|
| `*.combo.npy` | 2914 | 3401 | 20110104 ~ 20121231 (487 天) |

**业务含义**:
- 任何用 pwang 因子在 160 上跑回测, 时间窗包含 2011-2012 会拿到全 NaN
- 默认 `ops check` long_backtest 是 2015 起算, 不直接受影响
- 但**任何更长 rolling window / 早期 OOS 检验**两机会得到不同结果
- 41 个文件批量缺 = 单次 build 漏跑两年, 不是按文件零散漏

**怀疑根因**: 160 上 pwang 这套 module 加入 build 链路时, build_cc 没回填 2011-2012 段。147 当时已经 build 过完整历史。需 pwang 或 wbai 复跑 build_cc 把 160 这块补齐。

### 4.2 ⚠ 真 schema 漂移: `Basedata/st.npy` 跨机 dtype 不同

```
160: int8     (file_size 整除 N=5484)
147: float64  (file_size 整除 N×8)
```

**同一个文件名, 不同存储 dtype**, 跨机读出来类型不同, 后续比较 / NaN 处理走完全不同分支。

**业务含义**:
- ST 标记是 universe 过滤 + 因子 mask 的核心依据
- int8 `1/0` 模型 vs float64 `1.0/0.0/NaN` 模型, 缺失值语义不一样 (int8 没有 NaN 概念)
- **跟代码漂移报告里 `umgr_all.py` / `Universe.so vs .py` 实现差异直接相关**, 应该一起处理

**应进** [`2026-06-06-gsim-code-drift-three-sites.md`](2026-06-06-gsim-code-drift-three-sites.md) **的影响范围**: 之前那份报告假设代码差异**可能**导致数据差异, 现在有直接证据。

### 4.3 中等: 财务 / AI 盈利预测类 `nan_diff` (异步 build 固有特性)

**137 个文件**集中在以下目录:

| 目录 | 文件数 |
|---|---|
| `income_statement_fore_annual` | 42 |
| `income_statement_fore_quarter` | 32 |
| `financial_summary_fore_annual` | 22 |
| `financial_summary_fore_quarter` | 12 |
| `revenue_forecast_annual` | 2 |
| `revenue_forecast_quarter` | 2 |
| `DmgrPwang_industry_*` (sum_diff) | 19 |
| `Dpva` / `Dpvb` | 14 |

**怀疑根因**: 这些都源自 datayes 的 AI 盈利预测 + 财务一致预期, **每天都在更新 forecast**。两台机器各跑各的 build_cc, 捕获不同时刻的 datayes snapshot 是预期内的, 体现为 nan_count 微差和 sum 微差。

**业务含义**:
- **不是 bug, 是异步 build 的固有特性**
- 但意味着"两机器跑同一回测产出同样 PNL"这个隐含假设**对动态更新源头的 feature 不成立**
- 因子用 fore_* 数据时, 必须假设 "在哪台机器跑就用哪台的 snapshot", 不能跨机 reproduce

### 4.4 单点: `Basedata/cap.npy` / `capfree.npy`

```
都在 idx=3616 (20241105) 一天, NaN 数差 1 (391 vs 390)
```

某只股票在 2024-11-05 的市值状态在两边算出来不一致 (ST 切换 / 退市 / 复牌 等)。影响极小, 但说明 Basedata 的 build 流程在边缘日子可能跨机不一致。

### 4.5 误报: `Basedata/status.npy` sum_diff

```
idx=95 (20100506) 一天, max_rel_err=1.12e-2
sum_a = 8.73e-321,  sum_b = 8.83e-321 (denormalized double)
```

status 字段可能是位编码 status code 被读成 float64, 数值上看 1% 相对误差, 实际只是 1 bit 翻转 (单只股票状态变化)。**可忽略**, v2 应该对这类 status-encoded 文件特殊处理或跳过。

## 5. 跟代码漂移事件的关联

| 数据症状 | 对应代码漂移 |
|---|---|
| `Basedata/st.npy` dtype 不同 | `gsim/data/module/umgr_all.py` 差异 + Universe 实现路径不同 (160=.so / 147=.py + abi3) |
| pwang 缺 2 年 | 跟代码无关, 是 build_cc 调度问题 |
| fore_* nan_diff | 跟代码无关, 是 datayes 异步更新固有特性 |
| sum 微差 | **可能**跟 `Oputil.py` 1e-8 数值保护差异有关 (147 加了, 160 没有), 但是发生在归约层而不是相关性计算, 不直接关联 |

**好消息**: 即使代码层有 9 处共名 silent drift (含 `alpha_node.so` 6 倍体积差), 数据层 91% 文件依然等价。说明:
- 代码差异主要在 combo logic / 优化算子 / FeatureReader / Universe 加载入口, 这些不是 build_cc 主路径
- build_cc 用的 `source_ref/` 三地代码确认 byte-identical (160 vs 144), 跟 147 也只是文件子集差异, 共同文件全对
- **build_cc 出来的核心数据跨机是基本可信的**, 只有边缘 case (pwang 缺 build / Basedata schema 漂移 / forecast 异步) 有差

## 7. 工具 + 复现

工具: `scripts/data-audit/cc_fingerprint.py` + `cc_fingerprint_diff.py` + `README.md`

复现:
```bash
# 在 160 上
cd /home/wbai/gsim-ops/scripts/data-audit
python3 cc_fingerprint.py --out /tmp/fp_160.npz

# 在 147 上 (通过 rclone 中转, 147 SSH 不通)
rclone copy 39000:external-sync/scripts/cc_fingerprint.py /tmp/
python3 /tmp/cc_fingerprint.py --out /tmp/fp_147.npz
rclone copy /tmp/fp_147.npz /tmp/fp_147.skip.json /tmp/fp_147.summary.json \
    39000:external-sync/fingerprints/

# 回到 160 比对
rclone copy 39000:external-sync/fingerprints/ /tmp/147-fp/
python3 cc_fingerprint_diff.py /tmp/fp_160.npz /tmp/147-fp/fp_147.npz \
    --out /tmp/cc_diff_report.json
```
